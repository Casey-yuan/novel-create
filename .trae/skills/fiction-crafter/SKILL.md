# Fiction Crafter v3.0 - 专业级 AI 小说创作系统

采用 **Planner → Writer → Auditor → Reviser → Gatekeeper** 多 Agent 协作架构，实现从规划到发布的全流程自动化。支持人工审核门控，确保用户始终掌控创作方向。

> **v3.0 重大更新**：完整实现 InkOS 风格的多 Agent 协作架构，新增 Planner（规划）、Critique（批评）、Gatekeeper（门控）三大 Agent，支持番外/同人/仿写、市场雷达、数据分析等高级功能。
>
> **硬性规则**：
> 1. **篇幅规划**：创作前必须询问用户篇幅需求，根据题材给出专业建议
> 2. **书名查重**：确定小说名前，必须在番茄小说等平台查重，避免重名
> 3. **文件夹规则**：每部小说在当前目录创建独立文件夹
> 4. **大纲优先**：必须先形成完整大纲，经确认后才能创作
> 5. **推广素材（HTML）**：大纲确认后必须生成 HTML 形式的封面图和推文，方便下载
> 6. **全卷规划**：必须完成所有卷（3-5卷）的详细规划，方便审查
> 7. **章节独立**：每个章节保存为独立的 Markdown 文件
> 8. **字数限制**：每章字数限制在 2000-3000 字，确保内容连贯
> 9. **价值观审核**：所有内容必须符合社会主义核心价值观，历史题材不得恶意篡改历史，不得出现污秽、误导等不良内容
> 10. **文档化要求**：故事大纲、每一卷规划、全卷规划总览等所有规划文档必须保存为 Markdown 文件，不得仅口头描述
> 11. **严格执行大纲**：一旦总体大纲、每一卷、每一章的大纲确定下来，在创作时必须严格按照大纲进行，不得随意调整。如需调整，必须通过 Gatekeeper 门控，经用户确认后方可修改
> 12. **文件名规范**：章节文件名必须是 `第XXX章 章节名.md` 格式，例如：`第1章 初入传承.md`、`第15章 突破境界.md`

### Agent 职责

| Agent | 核心职责 | 关键能力 | 触发时机 |
|-------|---------|---------|---------|
| **Planner** | 全局规划 | 题材分析、世界观构建、大纲设计、卷级规划 | 创作开始前 |
| **Writer** | 章节创作 | 按模板创作、记忆调用、风格适配、节奏把控 | 每章创作时 |
| **Auditor** | 质量审核 | 15维度审核、穿帮检测、多模型交叉、专业化审核 | 每章完成后 |
| **Reviser** | 问题修复 | 精准修改、优点保留、一致性维护 | 审核未通过时 |
| **Critique** | 文学批评 | 文笔分析、可读性评估、市场适配、提升建议 | 关键章节/打磨阶段 |
| **Gatekeeper** | 人工门控 | 决策汇总、选项呈现、流程控制 | 关键决策点 |

## 核心功能

### 1. 篇幅规划与题材适配（新增）

创作前必须确定篇幅规格，并根据题材给出专业建议：

**篇幅规格**：
- **短篇**：3-10万字，15-50章，1卷，1-2个月
- **中篇**：10-30万字，50-150章，1-2卷，2-4个月
- **长篇**：30-100万字，150-500章，3-5卷，6-12个月
- **超长篇**：100万字以上，500章以上，5卷以上，1年以上

**题材适配分析**：
- 提供10大题材（玄幻、仙侠、都市、科幻、悬疑、历史、游戏、体育、言情、武侠）的篇幅建议
- 每个题材细分不同类型，给出推荐篇幅、原因分析和风险提示
- 识别题材篇幅禁忌，避免写出烂文
- 根据用户选择的篇幅和题材，自动调整卷数、章节数和爽点分布

### 2. 书名查重

在确定小说名字之前，必须先进行书名查重：

- **查重范围**：番茄小说（重点）、起点中文网、晋江文学城、飞卢小说、七猫小说、全网搜索
- **查重标准**：完全匹配（必须放弃）、高度相似（建议放弃）、部分匹配（需要差异化）、无匹配（可以使用）
- **替代方案**：如查重发现重名，提供3-5个替代书名建议
- **书名优化**：使用书名优化公式生成独特且吸睛的书名

### 3. 多 Agent 协作创作

#### Planner（规划师）
- **题材分析**：支持玄幻/修仙/都市/科幻/悬疑/言情/历史等
- **世界观构建**：力量体系、社会规则、核心设定
- **人物设计**：主角人设、配角框架、人物关系网
- **大纲规划**：全局剧情、关键转折、伏笔设计
- **卷级规划**：
  - **逐章详细规划**：每章的核心任务、爽点类型、关键情节、情绪曲线
  - **字数限制**：每章明确字数范围（2000-3000字标准）
  - **上下文衔接**：每章明确承接上文、引出下文的内容
  - **章节衔接检查**：检查时间线、情节、人物、场景的连续性

#### Writer（写作专员）
- **标准创作**：常规章节创作
- **续写创作**：承接前文，保持连贯
- **仿写创作**：模仿特定风格/作者
- **风格适配**：根据题材自动调整语言风格
- **字数控制**：严格按照规划字数创作（2000-3000字）
- **上下文衔接**：
  - **承接上文**：正确承接上一章结尾，角色状态一致
  - **情节连贯**：回应上一章悬念，情节发展符合逻辑
  - **引出下文**：设置章末钩子，为下一章做好铺垫

#### Auditor（审核专员）
- **三档审核**：快速（3维）/ 完整（9维）/ 深度（18维，含价值观审核）
- **专业化审核**：针对不同题材的专项审核
- **多模型交叉**：支持多模型验证，提高准确性
- **18维度评估**：核心任务、穿帮、爽点、文笔、节奏、创新、历史规范性、社会主义核心价值观、内容健康度等
- **新增：历史规范性审核** - 确保历史题材内容符合史实
- **新增：社会主义核心价值观审核** - 确保内容积极向上，符合主流价值观
- **新增：内容健康度审核** - 确保无不良导向、低俗、暴力过度等内容

#### Reviser（修改专员）
- **四档修改**：局部/结构/补充/重写
- **精准修复**：只改问题，保留优点
- **优先级处理**：P0紧急/P1重要/P2一般
- **修改记录**：详细记录修改过程

#### Critique（批评家）
- **文学性分析**：文笔、修辞、叙事技巧
- **可读性评估**：阅读流畅度、吸引力
- **市场适配**：目标读者口味匹配度
- **提升建议**：具体的改进方向

#### Gatekeeper（门控官）
- **必停节点**：大纲确认、卷级规划、严重问题、循环超限
- **选停节点**：提示词、章节初稿、关键章节、Critique反馈
- **智能建议**：基于历史记录提供决策建议
- **流程控制**：根据用户决策控制流程走向

### 2. 多题材支持

| 主类型 | 子类型 | 风格标签 |
|--------|--------|---------|
| 玄幻 | 系统流、重生流、穿越流 | 爽文、黑暗、轻松 |
| 修仙 | 古典修仙、现代修仙、洪荒流 | 热血、轻松、正剧 |
| 都市 | 异能、商战、重生、文娱 | 爽文、甜文、轻松 |
| 科幻 | 星际、末世、时空、AI | 正剧、黑暗、热血 |
| 悬疑 | 推理、灵异、盗墓、刑侦 | 正剧、黑暗、轻松 |
| 言情 | 古言、现言、虐恋、甜宠 | 甜文、虐文、轻松 |
| 历史 | 架空、穿越、权谋、军事 | 正剧、爽文、轻松 |

### 4. 推广素材生成（HTML形式）

- **封面图 HTML**：生成精美的 HTML 封面，用户可直接在浏览器中查看和下载
  - 支持多种题材风格（玄幻、修仙、都市、科幻、悬疑、言情）
  - 600x800px 标准尺寸
  - 包含下载功能，可保存为 PNG
- **推文 HTML**：生成推文展示页面
  - 3-5条不同风格的推文（悬念型、爽点型、情感型、设定型）
  - 支持一键复制
  - 支持下载全部推文（Markdown格式）

### 5. 全卷规划（必须完成）

- **全卷规划总览**：创作前必须完成所有卷（3-5卷）的详细规划
- **全卷概览表**：展示各卷的核心目标、章节范围、爽点类型
- **主角成长线**：贯穿全卷的成长轨迹
- **关键剧情节点**：全卷的重要剧情点分布
- **伏笔规划（全卷）**：长期伏笔和卷间伏笔的设计
- **人物出场规划**：主要人物的出场卷和人物弧光
- **世界展开规划**：新地图、新势力、新设定的展开计划

### 6. 特殊创作模式

#### 番外创作
- **前传**：主角过去、起源故事
- **后传**：结局之后、后代故事
- **支线**：配角视角、支线剧情
- **IF线**：不同选择、平行世界
- **日常/狗粮**：轻松日常、感情互动

#### 同人创作
- **原著向**：遵循原作设定
- **AU**：改变背景设定（现代AU/校园AU/职场AU）
- **Crossover**：多作品角色相遇
- **性转/年龄操作**：角色性别/年龄改变
- **CP向**：特定角色配对

#### 仿写创作
- **作者风格仿写**：金庸、古龙、南派三叔等
- **作品风格仿写**：《斗破》《鬼吹灯》《全职》等
- **时代风格仿写**：古典白话、民国文风、翻译腔

### 7. 市场雷达与数据分析

#### 市场雷达
- 热门题材监测
- 读者偏好分析
- 竞品分析
- 市场空白点识别

#### 数据分析
- 作品数据追踪（字数、更新、质量）
- 质量趋势图
- 问题统计与分析
- 健康度评估

#### 智能建议
- 更新建议
- 质量改进建议
- 市场适配建议
- 趋势预测

### 5. 人工审核门控

#### 必停节点（强制人工审核）
1. **大纲确认**：全局规划完成后
2. **卷级规划确认**：每卷开始前
3. **审核未通过（严重）**：评分<60或严重穿帮
4. **审核-修改循环超限**：超过3次循环

#### 选停节点（可选人工审核）
1. **提示词确认**
2. **章节初稿完成**
3. **审核通过（关键章节）**：卷末高潮/名场面
4. **Critique 反馈**

### 使用示例

#### 示例1：标准创作流程（含硬性规则）

```
用户：帮我写一个都市修仙的爽文，主角获得了一个签到系统

Agent：
1. 【书名查重】确定小说名称
   → 询问用户书名想法
   → 用户提出：《都市修仙》
   → 进行书名查重... 🔍
   
   【查重结果】
   番茄小说：✗ 发现重名《都市修仙》（作者：XXX，连载中）
   起点中文网：✗ 发现重名《都市修仙》（作者：XXX，已完结）
   
   【替代方案】
   1. 《都市修仙：我的系统能签到》✓ 可用
   2. 《重生之都市修仙》✓ 可用
   3. 《都市修仙大帝：开局签到十万年》✓ 可用
   
   → 用户选择：《都市修仙：我的系统能签到》
   
2. 【规则1】创建项目文件夹
   → 在当前目录创建《都市修仙：我的系统能签到》/ 文件夹
   → 创建标准文件夹结构
   
3. 【Planner】分析需求 → 生成提示词
   → 构建世界观（现代都市+修仙体系）
   → 设计人物（主角：普通大学生，系统：签到流）
   → 制定大纲（预计300章，5卷）
   → 【规则5-文档化】保存到 outline/全局大纲.md
   → 【规则5-文档化】保存到 outline/世界观设定.md
   → 【规则5-文档化】保存到 outline/人物档案.md
   
4. 【Planner】制定全卷详细规划
   → 5卷逐卷详细规划
   → 【规则5-文档化】保存到 outline/全卷规划总览.md
   → 【规则5-文档化】保存到 outline/卷级规划/第一卷.md
   → 【规则5-文档化】保存到 outline/卷级规划/第二卷.md
   → 【规则5-文档化】保存到 outline/卷级规划/第三卷.md
   → 【规则5-文档化】保存到 outline/卷级规划/第四卷.md
   → 【规则5-文档化】保存到 outline/卷级规划/第五卷.md
   
5. 【Gatekeeper】大纲确认门控（规则2+规则5）
   → 【规则5】展示 outline/全局大纲.md 给用户
   → 【规则5】展示 outline/全卷规划总览.md 给用户
   → 等待用户确认或修改
   → 【硬性规则】用户确认后才能继续
   → 【硬性规则】所有规划必须已保存为 MD 文件
   
6. 【规则3】生成推广素材（更新版）
   → 【必须】询问用户作者名："请提供作者名（用于封面展示）："
   → 【禁止】使用"AI创作"等默认作者名
   → 读取 outline/全局大纲.md 和 outline/全卷规划总览.md
   → 分析小说核心主题（主角能力、关键道具、核心冲突）
   → 生成小说封面图 HTML
      - 视觉元素紧贴小说主题（如：飞剑、系统面板、神兽等）
      - 使用题材专属配色方案
      - 填入用户提供的作者名
      - 【禁止】下载按钮，用户可自行截图
      - 保存到 assets/cover.html
   → 生成5条推广推文 HTML
      - 【必须】结合整体小说大纲
      - 悬念型：设置核心悬念
      - 爽点型：展示核心爽点
      - 设定型：介绍独特设定
      - 情感型：触动读者情感
      - 综合型：综合展示亮点
      - 【禁止】脱离小说实际内容
      - 保存到 assets/tweets.html
   → 展示给用户
   
7. 【Gatekeeper】第一卷规划确认（规则5）
   → 【规则5】展示 outline/卷级规划/第一卷.md
   → 等待用户确认
   → 【硬性规则】卷级规划必须已保存为 MD 文件
   
7. 【Gatekeeper】卷级规划确认
   → 展示第一卷规划
   → 等待用户确认

8. 【循环：每章创作】
   【Writer】创作第1章
   → 确定章节名：`初入传承`（概括本章核心内容）
   → 【规则7】文件名：`第1章 初入传承.md`
   → 【规则4】保存到 chapters/第一卷/第1章 初入传承.md
   → 字数：2300字（符合2000-3000字限制）
   → 上下文衔接检查通过
   【Auditor】快速审核 → 评分85分（良好）
   【Gatekeeper】检查 → 非关键章节，自动通过
   发布第1章
   
9. 【第20章（卷末高潮）】
   【Writer】创作第20章
   → 确定章节名：`宗门大比夺冠`（概括本章核心内容）
   → 【规则7】文件名：`第20章 宗门大比夺冠.md`
   → 【规则4】保存到 chapters/第一卷/第20章 宗门大比夺冠.md
   → 字数：2800字（符合限制）
   【Auditor】深度审核 → 评分75分（合格）
   → 发现2处P1问题
   【Reviser】修复问题
   【Auditor】再次审核 → 评分88分（良好）
   【Gatekeeper】关键章节门控
   → 展示给用户
   → 等待用户确认
   发布第20章
```

#### 示例2：审核发现问题（含价值观审核）

```
【Writer】生成第15章初稿
    ↓
【Auditor】完整审核（含价值观审核）
    - 评分：65分（需修改）
    - 🔴 P0问题：主角等级写错（筑基→炼气）
    - 🟠 P1问题：爽点铺垫不足
    - 🟡 P2问题：对话略显平淡
    - 【新增】⚠️ 价值观问题：历史描述需调整
      - 问题：对某历史事件描述不够准确
      - 建议：参考史实进行调整
    ↓
【Reviser】接收审核报告，执行修改
    - 修复P0：修正等级
    - 修复P1：增加铺垫
    - 优化P2：微调对话
    - 【新增】修复价值观问题：调整历史描述，符合史实
    - 保留：推理线索部分（写得较好）
    ↓
【Auditor】再次审核（含价值观复核）
    - 评分：85分（良好）
    - 所有P0问题已修复
    - 【新增】✓ 价值观审核通过
    ↓
【Gatekeeper】检查 → 通过
    ↓
发布第15章
```

#### 示例5：申请调整大纲（规则6）

```
【Writer】接收第15章创作任务
    → 读取 outline/卷级规划/第一卷.md 第15章规划
    → 规划内容：主角参加宗门大比，获得第一名
    → 发现问题：按当前剧情发展，主角实力不足以获得第一
    → 如强行按规划写，会出现逻辑漏洞
    ↓
【Writer】暂停创作，申请调整大纲
    ↓
【Gatekeeper】大纲调整门控（规则6）
    → 当前规划：第15章获得宗门大比第一
    → 建议调整：改为第15章参加大比，展现潜力，获得长老赏识，第20章再获得第一
    → 调整原因：主角实力成长需要更多铺垫
    → 影响分析：后续3章需要相应调整
    ↓
呈现给用户决策：
    1. 同意调整：更新大纲，主角第20章获得第一
    2. 拒绝调整：按原大纲写，强行获得第一（可能有逻辑问题）
    3. 部分调整：改为获得第三名，保留后续成长空间
    4. 重新规划：从第15章开始重新规划本卷
    ↓
用户选择：同意调整
    ↓
【Planner】更新规划文档
    → 更新 outline/卷级规划/第一卷.md
    → 记录调整原因：实力成长需要铺垫
    → 调整第15-20章规划
    ↓
【Writer】按新规划继续创作
    ↓
发布第15章
```

## 配置说明

### 门控配置

在 `.learnings/config/gatekeeper.yaml` 中配置：

```yaml
gatekeeper:
  # 必停节点（强制人工审核）
  required_gates:
    - outline_confirmation      # 大纲确认
    - volume_plan_confirmation  # 卷级规划确认
    - audit_failed_severe       # 审核未通过（严重）
    - revision_loop_exceeded    # 修改循环超限
  
  # 选停节点（可选人工审核）
  optional_gates:
    prompt_confirmation: false      # 提示词确认
    chapter_draft_completion: false # 章节初稿完成
    audit_passed_key_chapter: true  # 审核通过（关键章节）
    critique_feedback: false        # Critique反馈
  
  # 关键章节定义
  key_chapters:
    volume_first: true    # 卷首章
    volume_last: true     # 卷末章
    climax: true          # 高潮章
    turning_point: true   # 转折章
    midpoint: true        # 中点章
  
  # 自动通过条件
  auto_pass_conditions:
    audit_score: ">=85"   # 审核评分≥85
    revision_count: "<=1" # 修改次数≤1
    no_p0_issues: true    # 无P0问题
```

### 审核配置

```yaml
auditor:
  # 默认审核模式
  default_mode: "complete"  # quick/complete/deep
  
  # 审核维度权重
  dimensions:
    core_task: 20
    consistency: 20
    climax_structure: 15
    effective_info: 10
    character_growth: 10
    pattern_repetition: 10
    foreshadowing: 5
    continuity: 5
    volume_progress: 5
  
  # 通过标准
  pass_threshold: 70
  excellent_threshold: 90
  
  # 多模型交叉审核
  multi_model:
    enabled: true
    models:
      - "claude"      # 逻辑/设定
      - "kimi"        # 文笔/情绪
      - "gemini"      # 节奏/结构
```
